# Trail Time Predict V0.2 开发需求文档

## 1. 文档目的

本项目当前已经完成初版 Demo，具备以下基础能力：

- 上传并解析多个 `.fit` 文件
- 上传并解析比赛路线 `.gpx` 文件
- 提取平路、上坡、下坡和疲劳相关能力
- 根据个人能力画像预测比赛完赛时间
- 提供本地 Python / Streamlit 版本

V0.2 的目标不是继续堆叠大量新功能，而是先解决两个核心问题：

1. 当前能力模型和预测模型是否足够稳定、可解释。
2. 历史比赛结果不能被简单视为固定真值，系统需要区分“个人能力”“当天状态”和“比赛条件”。

本版本应从“单点时间预测工具”升级为：

> 基于个人稳定能力、当前状态和比赛条件的概率型越野完赛时间预测工具。

---

# 2. 核心建模原则

## 2.1 不将历史比赛成绩视为绝对真值

同一个人在同一条路线上的成绩也可能不同，原因包括：

- 当天身体状态
- 睡眠和恢复情况
- 近期训练疲劳
- 天气和温度
- 泥泞、积水或冰雪
- 补给是否顺利
- 配速策略
- 伤病、生病
- 比赛投入程度

因此，不能简单使用：

```text
预测时间 - 实际时间 = 模型误差
```

历史比赛仅用于：

- 检查模型是否存在长期系统性偏差
- 判断实际成绩是否落入合理预测区间
- 分析上坡、平路、下坡和疲劳阶段的相对误差
- 估算某场比赛的发挥系数
- 对模型进行长期校准

---

## 2.2 将比赛预测拆成三层

### 第一层：个人稳定能力

相对稳定、通过多个 FIT 文件提取：

- 平路有氧能力
- 平路阈值能力
- 不同坡度的上坡能力
- 不同坡度的下坡能力
- 不同时间阶段的疲劳衰减
- 各能力样本量和可信度

### 第二层：当前状态

短期变化，由用户输入或未来从近期数据估算：

- 状态很好
- 状态正常
- 轻微疲劳
- 状态较差
- 生病或受伤

### 第三层：比赛条件

外部因素：

- 温度
- 湿度
- 高海拔
- 泥泞程度
- 路面技术难度
- 补给站停留
- 装备负重
- 夜间路段

最终模型：

```text
最终比赛时间
=
标准能力时间
× 当前状态系数
× 环境系数
× 路况系数
× 策略系数
+ 补给停留时间
```

# 6. 能力模型优化

## 6.1 平路能力

筛选相对平坦路段：

```text
-2% <= grade <= 2%
```

保留：

- 有氧配速
- 阈值配速
- 不同心率区间下的稳定配速
- 样本波动

建议使用：

- 加权中位数
- 或加权 60% 至 70% 分位数

不要直接取最快值。

活动权重建议：

```text
最近30天：1.00
31至90天：0.85
91至180天：0.65
180天以上：0.40
```

活动类型权重：

```text
比赛：1.00
专项训练：0.85
普通训练：0.65
恢复训练：0.35
```

---

## 6.2 上坡能力

保留坡度分档用于展示：

```text
1%至5%
5%至10%
10%至15%
15%以上
```

但预测时不要直接按离散档位取值，应使用连续插值。

示例：

```python
def interpolate_uphill_vam(
    grade: float,
    points: list[tuple[float, float]]
) -> float:
    ...
```

输入：

```text
坡度节点：
3% -> 350 m/h
7.5% -> 620 m/h
12.5% -> 540 m/h
18% -> 430 m/h
```

当坡度为 11.2% 时，通过插值得到对应 VAM。

低坡度时不要只依赖 VAM。

建议：

```text
1%至5%：
主要参考水平速度和坡度配速

5%至10%：
水平速度与 VAM 混合

10%以上：
主要参考 VAM
```

---

## 6.3 下坡能力

下坡同样使用连续曲线，不使用固定坡度档位直接跳变。

下坡能力至少保留：

- 水平速度
- 垂直下降速度
- 坡度
- 样本持续时间
- 前期和后期能力差异

未来可加入技术难度，目前先由用户手动设置。

# 7. 疲劳模型优化

## 7.1 不再使用整体水平速度判断疲劳

错误示例：

```text
后半程平均速度 / 前半程平均速度
```

原因：

前后半程地形可能完全不同，容易把赛道变化误判为疲劳。

---

## 7.2 使用地形归一化疲劳

分别计算：

- 平路配速衰减
- 上坡 VAM 衰减
- 下坡速度衰减

建议结构：

```json
{
  "fatigue": {
    "flat": [
      {"hour": 0, "factor": 1.00},
      {"hour": 3, "factor": 0.96},
      {"hour": 5, "factor": 0.89},
      {"hour": 8, "factor": 0.79}
    ],
    "uphill": [
      {"hour": 0, "factor": 1.00},
      {"hour": 3, "factor": 0.94},
      {"hour": 5, "factor": 0.86},
      {"hour": 8, "factor": 0.74}
    ],
    "downhill": [
      {"hour": 0, "factor": 1.00},
      {"hour": 3, "factor": 0.97},
      {"hour": 5, "factor": 0.88},
      {"hour": 8, "factor": 0.72}
    ]
  }
}
```

---

## 7.3 疲劳曲线使用连续插值

不要使用：

```text
0至3小时一个值
3至5小时一个值
5小时以后一个值
```

避免在 2:59 和 3:01 出现突变。

实现：

```python
def interpolate_fatigue(
    elapsed_hours: float,
    fatigue_points: list[tuple[float, float]]
) -> float:
    ...
```

# 8. 能力可信度

每个能力值都必须附带可信度。

## 8.1 可信度考虑因素

- 样本数量
- 样本总时长
- 样本总距离
- 样本累计爬升或下降
- 样本时间跨度
- 数据质量
- 是否使用个人真实数据
- 是否使用系统默认值

---

## 8.2 简单评分规则

可以先用规则模型：

```text
无个人样本，使用默认值：0.20

个人样本时长小于10分钟：0.35
10至30分钟：0.50
30至90分钟：0.70
90分钟以上：0.85

近期样本充分且波动较小：最高0.95
```

结果展示：

```text
10%至15%上坡能力：
540 m/h
可信度：0.82
样本数：13
样本时长：74分钟
累计爬升：1030米
```

# 9. 标准能力时间

系统首先计算一个不考虑当天状态和特殊比赛条件的时间：

```text
标准能力时间
```

定义：

> 在身体健康、正常恢复、合理配速、正常天气、正常路况和正常补给情况下，该运动员完成该路线的预计移动时间。

该时间用于表示稳定能力，而不是某一天的实际必然成绩。

# 10. 状态和比赛条件修正

## 10.1 当前状态系数

第一版由用户手动选择：

```text
状态很好：0.97
状态正常：1.00
轻微疲劳：1.04
状态较差：1.08
生病或伤病：1.15
```

实现为配置项，不要硬编码在业务逻辑中。

---

## 10.2 技术难度系数

第一版由用户手动选择：

```text
0 公路或平整林道
1 简单山径
2 普通山径
3 中等技术
4 高技术
```

技术难度对不同地形的影响不同：

```text
对平路影响较小
对上坡影响中等
对下坡影响最大
```

建议分别设置：

```python
TECHNICAL_FACTORS = {
    0: {"flat": 1.00, "uphill": 1.00, "downhill": 1.00},
    1: {"flat": 1.01, "uphill": 1.02, "downhill": 1.04},
    2: {"flat": 1.03, "uphill": 1.04, "downhill": 1.09},
    3: {"flat": 1.05, "uphill": 1.07, "downhill": 1.16},
    4: {"flat": 1.08, "uphill": 1.12, "downhill": 1.28}
}
```

这些参数后续通过历史数据校准。

# 11. 概率预测

## 11.1 不只输出单点时间

输出至少包括：

```text
最快合理时间
中位预测时间
保守预测时间
预测可信度
```

示例：

```text
标准能力时间：7:35
状态和环境修正后中位时间：7:52
最快合理时间：7:24
保守预测时间：8:31
预测可信度：中等
```

---

## 11.2 Monte Carlo 模拟

第一版使用 Monte Carlo 生成时间分布。

每次模拟随机采样：

```text
平路能力
上坡能力
下坡能力
疲劳曲线
状态系数
技术难度系数
GPX高程误差
```

示例：

```python
uphill_factor = random.normalvariate(1.0, uphill_sigma)
downhill_factor = random.normalvariate(1.0, downhill_sigma)
fatigue_factor = random.normalvariate(1.0, fatigue_sigma)
condition_factor = random.normalvariate(base_condition, condition_sigma)
```

建议模拟次数：

```text
默认 3000 次
可配置 1000 至 10000 次
```

结果取：

```text
P10：最快合理时间
P50：中位时间
P90：保守时间
```

标准差根据能力可信度决定：

```text
高可信度：较小波动
低可信度：较大波动
系统默认值：最大波动
```

# 12. 历史比赛解释与模型校准

模块名称：

```text
Historical Race Interpretation and Calibration
```

不要命名为简单的“预测准确率测试”。

---

## 12.1 输入

- 用于建立能力画像的历史 FIT
- 某场已完成比赛 GPX
- 该比赛实际 FIT
- 用户对当天状态和环境的补充信息，可选

---

## 12.2 输出

```text
标准能力预测
考虑用户填写状态后的预测
实际比赛时间
实际成绩在预测分布中的百分位
估算发挥系数
上坡相对表现
平路相对表现
下坡相对表现
前中后程表现
可能的异常原因
```

示例：

```text
标准能力时间：7:38
当天条件修正后中位时间：7:57
实际时间：8:08

实际成绩位于预测分布 P68
属于正常发挥范围

估算整体发挥系数：1.023

上坡：比个人标准慢3%
平路：比个人标准慢9%
下坡：比个人标准快2%
5小时后疲劳：高于个人历史平均
```

---

## 12.3 校准原则

单场比赛不得直接修改个人能力。

只有满足以下条件才允许调整：

- 多场比赛出现同方向偏差
- 相同地形持续偏差
- 偏差超过设定阈值
- 数据质量良好
- 用户未标记明显异常状态

例如：

```text
连续3场比赛：
10%至15%上坡预测均比实际快8%以上
```

可以建议：

```text
将该坡度能力曲线下调4%至6%
```

不要一次性完全修正。

# 13. 数据质量诊断

## 13.1 FIT 质量检查

检查：

- 时间戳是否连续
- 是否存在大量零速度
- 是否存在 GPS 跳点
- 是否存在异常瞬时速度
- 海拔是否跳变
- 距离是否递增
- 心率是否大量缺失
- 活动是否过短
- 是否可能为徒步或非跑步活动

输出：

```text
数据质量：高 / 中 / 低
问题列表
是否建议用于能力建模
```

---

## 13.2 GPX 质量检查

检查：

- 原始点数
- 平均点间距
- 海拔缺失率
- 最大单点高度跳变
- 原始累计爬升
- 平滑后累计爬升
- 不同重采样尺度的爬升差异

建议同时计算：

```text
50米重采样
100米重采样
200米重采样
```

如果累计爬升差异较大：

```text
提示：该GPX海拔噪声较高，预测区间已自动扩大。
```

# 18. 目标时间反推

实现一个简单版本。

用户输入：

```text

```

系统输出：

```text
当前中位预测
与目标差距
主要时间损耗来源
需要提升的能力
```

示例：

```text

当前中位预测：8小时27分
差距：27分钟

主要差距：
1. 15%以上上坡：14分钟
2. 5小时后下坡衰减：9分钟

达到目标的建议：
- 陡坡VAM从430提升到470 m/h
- 5小时后下坡保留率从0.76提升到0.82
- CP总停留控制在15分钟内
```

本功能先使用规则反推，不需要机器学习。

# 23. Codex 执行要求

请先阅读现有仓库代码，不要完全重写项目。

执行原则：

1. 保留现有可以工作的 FIT、GPX 解析逻辑。
2. 优先通过重构和新增模块实现功能。
3. 每完成一个 Phase，保证现有 Demo 仍可运行。
4. 所有新增经验参数放入配置文件。
5. 所有核心数据结构增加类型标注。
6. 关键算法增加 docstring。
7. 为新增功能编写测试。
8. 不引入数据库。
9. 不引入 Java 或其他后端。
10. 保持本地 Python 和 Streamlit 运行方式。
11. 对缺失数据提供安全降级，不允许程序直接崩溃。
12. 输出结果必须明确区分：
    - 标准能力时间
    - 状态和环境修正
    - 移动时间
    - 补给停留时间
    - 最终预测区间

请先完成 Phase 1，再继续 Phase 2，不要一次性实现全部功能。
